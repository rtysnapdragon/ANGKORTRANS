Normal Forms

Database normalization applied to each module

1NF — First Normal Form

Atomic values, no repeating groups

Applied across all tables. Each column holds one value per row.

  • No comma-separated product IDs in orders
  • No multi-value phone fields in customers
  • Each order item is its own row in order_items
  • Tracking events split into separate rows
2NF — Second Normal Form

No partial dependencies on composite keys

Applies to tables with composite PKs like order_items.

  • product_name removed from order_items — lives in products only
  • customer_email not stored in orders table
  • warehouse_name not duplicated in inventory
  • Supplier info lives in suppliers, not purchase_orders
3NF — Third Normal Form

No transitive dependencies

Non-key attributes depend only on the primary key.

  • Category details extracted to categories table
  • Carrier info in its own carriers reference table
  • Warehouse city/region not derived from zip_code
  • Tax rate stored by region, not duplicated per order
BCNF — Boyce-Codd NF

Every determinant is a candidate key

Ensures no anomalies in edge cases beyond 3NF.

  • SKU uniquely identifies product — no alternate FD paths
  • Tracking number determines shipment uniquely
  • Product–warehouse pair is the composite key for inventory
  • Verified no overlapping candidate keys exist
